# Workflow: ATDD (Acceptance Test Driven Development)

## Purpose
Синхронизация команды вокруг приёмочных критериев до начала разработки. Каждая user story получает автоматизированные приёмочные тесты.

## Input
- User story с критериями приёмки (Acceptance Criteria)
- Понимание проблемной области (домена)

## Steps

### Step 1: Three Amigos (Три друга)
- Действия: Product Owner + разработчик + тестировщик обсуждают историю
- Результат: единое понимание, уточнение AC
- Артефакт: уточнённые критерии, вопросы к команде

### Step 2: Specify Examples (Спецификация примеров)
- Действия: описать 3-5 конкретных примеров, покрывающих счастливый путь, ошибки и граничные случаи
- Формат: Given-When-Then (Gherkin)
- Результат: feature-файлы (*.feature)

### Step 3: Automate Tests (Автоматизация тестов)
- Действия: реализовать step definitions для сценариев
- Инструменты: Cucumber, SpecFlow, Behave
- Результат: запускаемые приёмочные тесты (красные)

### Step 4: Implement (Реализация)
- Действия: писать продакшен-код, пока тесты не станут зелёными
- Правила: минимальный код для прохождения теста
- Результат: работающий функционал, зелёные тесты

### Step 5: Refactor (Рефакторинг)
- Действия: улучшить код, тесты остаются зелёными
- Результат: чистый код, полное покрытие приёмочных сценариев

## Output
- Feature-файлы с приёмочными сценариями (Given-When-Then)
- Автоматизированные тесты, интегрированные в CI
- Единое понимание требований всей командой
