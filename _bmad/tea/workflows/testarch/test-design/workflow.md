# Workflow: Тест-дизайн (Test Design)

## Purpose
Создание набора тестов, обеспечивающих максимальное покрытие при минимальном количестве тест-кейсов.

## Input
- Функциональные требования / user stories
- Приёмочные критерии (Acceptance Criteria)
- Понимание рисков

## Steps

### Step 1: Analyze (Анализ)
- Действия: разбить функционал на атомарные unit-сценарии
- Выявить входные и выходные данные, состояния, переходы
- Результат: функциональная декомпозиция

### Step 2: Select Technique (Выбор техники)
- **Equivalence Partitioning** — деление входных данных на классы эквивалентности
- **Boundary Value Analysis** — тестирование границ каждого класса
- **Decision Table** — комбинации условий (business rules)
- **State Transition** — изменение состояний системы
- **Pairwise Testing** — покрытие комбинаций параметров
- Результат: выбранные техники и обоснование

### Step 3: Design Cases (Разработка кейсов)
- Действия: применить выбранные техники; описать тест-кейсы
- Формат: Given-When-Then или шаги + ожидаемый результат
- Результат: набор тест-кейсов

### Step 4: Peer Review (Ревью)
- Действия: коллеги проверить полноту, корректность, читаемость
- Критерии: нет дублирования, граничные случаи, понятные шаги
- Результат: проверенные и утверждённые тест-кейсы

### Step 5: Prioritize (Приоритизация)
- Действия: назначить приоритет (P0-P3) по критичности и частоте
- P0: smoke (должны быть зелёными всегда)
- P1: critical (основная функциональность)
- P2-P3: расширенное покрытие
- Результат: приоритезированный тест-план

## Output
- Набор тест-кейсов с техниками тест-дизайна
- Приоритезированный тест-план
- Документация: обоснование выбора техник и покрытия
