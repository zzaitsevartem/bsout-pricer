# Workflow: Проектирование архитектуры решения

## Назначение
Спроектировать техническую архитектуру продукта: архитектурные паттерны, компоненты, потоки данных и нефункциональные требования.

## Входные данные
- PRD
- Результаты технического исследования
- UX-дизайн (навигация, экраны)
- Существующая архитектура (для brownfield)

## Участники
- Архитектор (ведущий)
- Tech Lead
- Разработчики ключевых модулей
- DevOps-инженер

## Шаги

### 1. Анализ требований
- Выделить архитектурно значимые требования (ASR)
- Зафиксировать нефункциональные требования (NFR)
- Определить качественные атрибуты: scalability, reliability, security

### 2. Выбор архитектурного стиля
- Определить стиль: микросервисы, modular monolith, serverless
- Обосновать выбор по критериям
- Согласовать с командой

### 3. Проектирование компонентов
- Выделить bounded contexts и модули
- Спроектировать API-контракты (синхронные/асинхронные)
- Определить схему данных и модель хранения

### 4. Потоки данных и интеграции
- Построить диаграмму потоков данных
- Описать интеграции с внешними системами
- Определить стратегию обработки ошибок

### 5. Документирование и ревью
- Оформить ADR (Architecture Decision Records)
- Подготовить C4-диаграммы (уровни контекст/контейнеры/компоненты)
- Провести архитектурное ревью

## Выходные данные
- Architecture Document
- ADR по ключевым решениям
- C4-диаграммы
- Risk Register (технические риски)
